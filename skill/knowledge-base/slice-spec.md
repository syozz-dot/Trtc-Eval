## 一、什么是 Slice

Slice(原子能力)是 TRTC AI 知识库的最小知识单元。每个 slice 描述一个**独立的功能点**,包含:这个功能是什么、怎么用、常见的坑、出了问题怎么排查。
Slice(原子能力)是 TRTC AI 知识库的最小知识单元。每个 slice 描述一个**独立的功能点**,包含:这个功能是什么、怎么用、常见的坑、出了问题怎么排查。

**类比**:如果 TRTC SDK 是一台车,slice 就是驾驶手册里的一个章节——"怎么启动"、"怎么倒车"、"怎么用定速巡航",每一章独立可读,合在一起就是完整手册。

**最终读者是 AI**:slice 文档会被 AI 直接读取并用于生成代码。这意味着:
- 任何模糊词("应该""大概""一般来说")都会被 AI 解释为"可选" → 跳过
- 任何遗漏的反例,AI 都会用训练数据里的错误模式去补
- 任何不可机械验证的规则,AI 都可能"凑字符串"通过

→ 所以本 spec 用极端强势的语言,是为了让产出的 slice 也具有同样的硬约束力。

### 强制拆分判断流程(任何一步不可跳过)

新建 slice 前,**必须**按顺序回答以下 4 个问题。任何一个问题答不出 → 不允许新建 slice。

#### 问题 1:能不能一次讲清楚?

**判断标准**:在 PR 描述中粘贴一份 ≤500 字的"客户问这个问题时的一次性回答 demo"。
- 超过 500 字 → split,本 slice 太大
- 不到 50 字 → 太小,合并到相关 slice
- 恰好够讲清原理 + 关键 API + 常见错误码 → ✅ 合格粒度

✅ **正例**:
- "多端登录老是互踢怎么办?" → 讲策略配置 + 回调处理 + 常见错误码,300 字搞定 ✅
- "消息发不出去" → 讲发送流程 + 权限检查 + 错误码,400 字搞定 ✅

❌ **反例**:
- "消息功能怎么做?" → 发送+接收+历史+撤回+已读+推送... 至少 3000 字 ❌(太大,split)

## 三、Slice 两层架构:主线 + 反馈

| 层级 | 名称 | 来源 | 用途 |
|------|------|------|------|
| 🅰️ **主线 Slice** | 骨架 | 按 SDK 能力域系统规划(来自官方文档+研发经验) | 保证每个核心功能都有覆盖 |
| 🅰️ **主线 Slice** | 骨架 | 按 SDK 能力域系统规划(来自官方文档+研发经验) | 保证每个核心功能都有覆盖 |

### 两者的关系
- **主线是骨架**:每个开发者都要走的路(初始化 → 登录 → 发消息 → ...)
- **反馈是血肉**:开发者最容易摔跤的坑(互踢死循环、推送不到达、...)
- 反馈 slice 可以是主线 slice 的深度补充,也可以是全新的边缘场景

###### 3️⃣ ✅ 正例(iOS)
```markdown
## 平台特有注意事项

- **`AnyCancellable` 必须存为实例属性,不能存为局部变量** —— 局部变量出
  作用域后 cancellable 被释放,sink 闭包永远不会触发。
  必须做:声明 `private var cancellables = Set<AnyCancellable>()` 实例属性,
  调用 `.store(in: &cancellables)`。

- **`sink { [weak self] }` 必须显式写 `[weak self]`** —— 不写会导致
  ViewModel 与 publisher 循环引用,VC 销毁后 ViewModel 不释放。
  必须做:所有 `.sink { }` 闭包第一行都加 `[weak self]`。

- **首次申请麦克风权限的弹窗在异步线程触发会被忽略** —— 必须在主线程
  调用 `applyForSeat`,否则系统不展示权限弹窗、SDK 报权限拒绝。
  必须做:`DispatchQueue.main.async { applyForSeat() }`。
```
